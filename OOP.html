What is OOP?
The paradigm, the philosophy, the why
Definition
Object-Oriented Programming (OOP) is a programming paradigm that organizes software design around objects — bundling data (attributes) and behavior (methods) together — rather than around functions and logic alone.

Programming Paradigms at a Glance

ParadigmCore IdeaExample
ProceduralStep-by-step instructions, functionsC, Pascal
FunctionalPure functions, immutable dataHaskell, Erlang
OOPObjects with state and behaviorJava, Python, C++
DeclarativeDescribe what, not howSQL, HTML

Why OOP Exists

As programs grew larger in the 1970s–80s, procedural code became an unmanageable tangle of global variables and functions. OOP was born to solve:

  • Complexity — model the real world as objects
  • Reusability — write once, use everywhere via inheritance
  • Maintainability — changes inside a class don't break outside code
  • Collaboration — teams can work on different classes independently

Classes & Objects
The blueprint and the thing itself
🏠 Class = Blueprint

A template/definition. Describes what data and behaviors instances will have. No memory allocated.

🏡 Object = Instance

A real, living thing created from the class. Occupies memory. You can have millions of objects from one class.

Analogy
A class is like an architectural blueprint for a house. An object is an actual house built from that blueprint. Many houses can be built from the same blueprint, each with different paint colors and furniture.
Python
# CLASS DEFINITION (blueprint)
class Dog:
    # Class body goes here
    pass

# OBJECT CREATION (instantiation)
my_dog = Dog()       # my_dog is now an OBJECT (instance of Dog)
your_dog = Dog()     # another separate object

# They are different objects!
print(my_dog is your_dog)  # False — different instances

# Check type
print(type(my_dog))          # <class '__main__.Dog'>
print(isinstance(my_dog, Dog))  # True

Everything in Python is an Object

Python
# Even basic types are objects in Python!
x = 42
print(type(x))        # <class 'int'>
print(x.bit_length()) # 6  — calling a method on int!

name = "Alice"
print(name.upper())    # ALICE — str is a class too

Attributes & Methods
State and behavior
TermWhat It IsAnalogy
Attribute / FieldVariable stored inside an object (state/data)Dog's name, breed, age
MethodFunction defined inside a class (behavior)Dog's bark(), fetch(), eat()
selfReference to the current instance (Python)"me" — this specific dog
Python
class Dog:
    # ── CLASS ATTRIBUTE ── shared by ALL instances
    species = "Canis lupus familiaris"

    def __init__(self, name, breed, age):
        # ── INSTANCE ATTRIBUTES ── unique per object
        self.name = name
        self.breed = breed
        self.age = age
        self.is_hungry = True

    # ── INSTANCE METHODS ── operate on self
    def bark(self):
        return f"{self.name} says: Woof!"

    def eat(self, food):
        self.is_hungry = False
        return f"{self.name} eats {food} 🍖"

    # ── CLASS METHOD ── operates on the class itself
    @classmethod
    def get_species(cls):
        return cls.species

    # ── STATIC METHOD ── no self or cls needed
    @staticmethod
    def is_dog_friendly(breed):
        friendly = ["Golden Retriever", "Labrador"]
        return breed in friendly

# ── Using the class ──
rex = Dog("Rex", "German Shepherd", 3)
print(rex.bark())          # Rex says: Woof!
print(rex.eat("bone"))     # Rex eats bone 🍖
print(Dog.get_species())   # Canis lupus familiaris
print(Dog.is_dog_friendly("Labrador"))  # True
Key Rule
Every instance method must have self as its first parameter — Python passes the instance automatically when you call rex.bark().

Constructors & Magic Methods
Object lifecycle and special behaviors

A constructor is a special method called automatically when an object is created. In Python it's __init__. In Java/C++ it's a method with the same name as the class.

Python — Magic / Dunder Methods
class BankAccount:
    def __init__(self, owner, balance=0):
        """Constructor — runs on object creation"""
        self.owner = owner
        self.balance = balance

    def __str__(self):
        """Called by print() or str()"""
        return f"Account({self.owner}: ${self.balance})"

    def __repr__(self):
        """Official string representation (for developers)"""
        return f"BankAccount('{self.owner}', {self.balance})"

    def __eq__(self, other):
        """Define == behavior"""
        return self.owner == other.owner

    def __add__(self, amount):
        """Define + operator: account + 100"""
        return BankAccount(self.owner, self.balance + amount)

    def __len__(self):
        """len(account) returns balance"""
        return self.balance

    def __del__(self):
        """Destructor — called when object is garbage collected"""
        print(f"Account {self.owner} closed")

acc = BankAccount("Alice", 500)
print(acc)            # Account(Alice: $500)
acc2 = acc + 200      # uses __add__
print(len(acc2))      # 700

Encapsulation
Bundle data, control access
Definition
Encapsulation is the bundling of data and the methods that operate on that data into a single unit (the class), while restricting direct access to some of the object's components. This prevents accidental modification and protects the object's internal state.

Access Modifiers

Python ConventionMeaningJava Equivalent
name (no prefix)Public — accessible anywherepublic
_name (single underscore)Protected — convention: don't touch from outsideprotected
__name (double underscore)Private — name-mangled, hard to access outsideprivate
Python — Encapsulation in Action
class BankAccount:
    def __init__(self, owner, balance):
        self.owner = owner          # public
        self._bank_code = "XYZ"   # protected (convention)
        self.__balance = balance   # private (name-mangled)

    # GETTER — controlled read access
    def get_balance(self):
        return self.__balance

    # SETTER — controlled write access with validation
    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("Deposit must be positive!")
        self.__balance += amount

    def withdraw(self, amount):
        if amount > self.__balance:
            raise ValueError("Insufficient funds!")
        self.__balance -= amount

acc = BankAccount("Alice", 1000)
print(acc.get_balance())   # 1000 ✅
# acc.__balance = 999999  # Won't work as expected!
# acc.__balance still returns AttributeError

# Python's name mangling: actual name is _BankAccount__balance
print(acc._BankAccount__balance)  # 1000 (accessible but discouraged)

Python Properties — Elegant Getters/Setters

Python — @property
class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius   # calls the setter below!

    @property
    def celsius(self):           # GETTER
        return self.__celsius

    @celsius.setter
    def celsius(self, value):   # SETTER with validation
        if value < -273.15:
            raise ValueError("Below absolute zero!")
        self.__celsius = value

    @property
    def fahrenheit(self):       # COMPUTED property — no setter needed
        return self.__celsius * 9/5 + 32

t = Temperature(100)
print(t.celsius)      # 100   (looks like attribute, runs getter)
print(t.fahrenheit)   # 212.0 (computed on the fly!)
t.celsius = 0          # runs setter — validates
# t.celsius = -999     # ❌ ValueError: Below absolute zero!
05
Inheritance
Build on what exists
Definition
Inheritance allows a class (child/subclass) to inherit attributes and methods from another class (parent/superclass), enabling code reuse and hierarchical relationships.
Animal ← Parent / Base class / \ Dog Cat ← Child / Derived classes / \ Bulldog Poodle ← Grandchild classes
Python — Inheritance
# ── PARENT CLASS ──
class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def breathe(self):
        return f"{self.name} breathes"

    def speak(self):         # will be overridden
        return "..."

    def __str__(self):
        return f"{self.__class__.__name__}({self.name})"

# ── CHILD CLASS ──
class Dog(Animal):         # inherits from Animal
    def __init__(self, name, age, breed):
        super().__init__(name, age)  # call parent constructor!
        self.breed = breed             # new attribute

    def speak(self):           # METHOD OVERRIDE
        return f"{self.name} says Woof!"

    def fetch(self):           # NEW method only in Dog
        return f"{self.name} fetches the ball!"

class Cat(Animal):
    def speak(self):
        return f"{self.name} says Meow!"

# ── Using inheritance ──
rex = Dog("Rex", 3, "Husky")
print(rex.breathe())  # inherited from Animal → "Rex breathes"
print(rex.speak())    # overridden → "Rex says Woof!"
print(rex.fetch())    # Dog-specific → "Rex fetches the ball!"

# Check inheritance
print(isinstance(rex, Dog))     # True
print(isinstance(rex, Animal))  # True! Dog IS-A Animal
print(issubclass(Dog, Animal))  # True

Multiple Inheritance (Python)

Python — Multiple Inheritance + MRO
class Flyable:
    def fly(self):
        return f"{self.name} is flying!"

class Swimmable:
    def swim(self):
        return f"{self.name} is swimming!"

class Duck(Animal, Flyable, Swimmable):  # multiple inheritance!
    def speak(self):
        return f"{self.name} says Quack!"

donald = Duck("Donald", 2)
print(donald.fly())    # Donald is flying!
print(donald.swim())   # Donald is swimming!

# Method Resolution Order (MRO) — Python's search order
print(Duck.__mro__)  # (Duck, Animal, Flyable, Swimmable, object)
⚠️ Diamond Problem
When two parent classes both inherit from the same grandparent, there's ambiguity about which version of a method to use. Python solves this with C3 linearization (MRO). Java avoids it by only allowing single class inheritance.

Polymorphism
"Many forms" — same interface, different behavior
Definition
Polymorphism (Greek: "many shapes") means that objects of different classes can be treated as objects of a common parent class. The same method call produces different results depending on the actual object type.

Types of Polymorphism

TypeDescriptionPython Support
Runtime (subtype)Method overriding via inheritance✅ Native
Duck Typing"If it quacks like a duck..." — no explicit type check✅ Native
Operator OverloadingSame operator, different behavior per type✅ via __dunder__
Method OverloadingSame name, different parameter signatures❌ Not native (use *args)
Python — Polymorphism
# ── Runtime Polymorphism ──
class Shape:
    def area(self): return 0
    def __str__(self): return f"{self.__class__.__name__}(area={self.area()})"

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return 3.14159 * self.r ** 2

class Rectangle(Shape):
    def __init__(self, w, h): self.w = w; self.h = h
    def area(self): return self.w * self.h

class Triangle(Shape):
    def __init__(self, b, h): self.b = b; self.h = h
    def area(self): return 0.5 * self.b * self.h

# 🎭 POLYMORPHISM: treat all shapes uniformly!
shapes = [Circle(5), Rectangle(4, 6), Triangle(3, 8)]

for shape in shapes:
    print(shape.area())  # Each calls ITS OWN area()
    # 78.53975  /  24  /  12.0

# ── Duck Typing ──
class PDFDocument:
    def render(self): return "Rendering PDF..."

class HTMLDocument:
    def render(self): return "Rendering HTML..."

def display(doc):    # doesn't care about type — just needs render()
    print(doc.render())

display(PDFDocument())   # Rendering PDF...
display(HTMLDocument())  # Rendering HTML...

Abstraction
Expose the essential, hide the complexity
Definition
Abstraction means showing only the relevant details to the user and hiding implementation complexity. You use a TV remote without knowing how it sends infrared signals — that complexity is abstracted away.
✅ What Users See

drive_car() — simple interface
send_email("hello") — clean API
list.sort() — just works

🔧 Hidden Complexity

Engine cylinders, fuel injection,
SMTP protocol, DNS lookup,
quicksort algorithm internals

Python — Abstract Base Classes (ABC)
from abc import ABC, abstractmethod

class PaymentProcessor(ABC):  # Abstract class — can't be instantiated!

    @abstractmethod
    def process_payment(self, amount):
        pass   # No implementation — subclasses MUST implement this

    @abstractmethod
    def refund(self, transaction_id):
        pass

    def get_fee(self):           # Concrete method — has implementation
        return 0.029             # 2.9% fee

# Concrete implementations
class StripeProcessor(PaymentProcessor):
    def process_payment(self, amount):
        return f"Stripe: charging ${amount} via card network..."
    def refund(self, tid):
        return f"Stripe: refunding transaction {tid}"

class PayPalProcessor(PaymentProcessor):
    def process_payment(self, amount):
        return f"PayPal: transferring ${amount} via PayPal balance..."
    def refund(self, tid):
        return f"PayPal: issuing refund for {tid}"

# PaymentProcessor()  # ❌ TypeError: Can't instantiate abstract class
p = StripeProcessor()
print(p.process_payment(100))  # works!

Interfaces & Protocols
Contracts without implementation

An interface is a pure contract — it declares what methods a class must have, without providing any implementation. Java has explicit interface keyword; Python uses ABCs or Protocols (typing).

Python — Protocol (Structural Subtyping)
from typing import Protocol

# Protocol = interface in Python typing
class Drawable(Protocol):
    def draw(self) -> str: ...
    def resize(self, factor: float) -> None: ...

class Circle:
    def draw(self) -> str: return "○"
    def resize(self, f): self.radius *= f

# Circle satisfies Drawable without explicitly inheriting it!
def render(item: Drawable):   # type-safe!
    print(item.draw())

Abstract Class vs Interface

FeatureAbstract ClassInterface
Has implementationYes (some methods)No (pure contract)
Has state/attributesYesNo (usually)
InheritanceSingle (Java)Multiple allowed
Purpose"IS-A" relationship"CAN-DO" relationship
Python keywordABCProtocol / ABC (all abstract)

Composition vs Inheritance
"Favor composition over inheritance" — GoF
Key Insight
Inheritance = "IS-A" (Dog IS-A Animal) → Use when a true parent-child relationship exists.
Composition = "HAS-A" (Car HAS-A Engine) → Use when you want to reuse behavior without being the thing.
Python — Composition
# ── BAD: Inheritance misuse ──
class Logger:
    def log(self, msg): print(f"[LOG] {msg}")

class UserService(Logger):  # ❌ UserService IS-A Logger? No!
    def create_user(self, name):
        self.log(f"Creating {name}")

# ── GOOD: Composition ──
class Logger:
    def log(self, msg): print(f"[LOG] {msg}")

class UserService:           # ✅ UserService HAS-A Logger
    def __init__(self):
        self.logger = Logger()  # composed inside

    def create_user(self, name):
        self.logger.log(f"Creating {name}")

# ── Real World: Car example ──
class Engine:
    def start(self): return "Vroom!"
    def stop(self): return "Shhhh..."

class GPS:
    def navigate(self, dest): return f"Navigating to {dest}"

class Car:                     # Car HAS-A Engine, HAS-A GPS
    def __init__(self, model):
        self.model = model
        self.engine = Engine()
        self.gps = GPS()

    def drive_to(self, dest):
        self.engine.start()
        return self.gps.navigate(dest)

my_car = Car("Tesla")
print(my_car.drive_to("Paris"))  # Navigating to Paris

SOLID Principles
The 5 laws of good OOP design
S — Single Responsibility Principle

A class should have only ONE reason to change. Don't bundle unrelated logic into one class.

Python
# ❌ BAD — Order does everything
class Order:
    def calculate_total(self): ...
    def save_to_db(self): ...      # DB logic here!?
    def send_email(self): ...       # Email logic too!?

# ✅ GOOD — Each class has one job
class Order:
    def calculate_total(self): ...

class OrderRepository:
    def save(self, order): ...

class OrderNotifier:
    def send_confirmation(self, order): ...
O — Open/Closed Principle

Open for extension, closed for modification. Add new behavior by adding new code, not changing existing code.

Python
# ❌ BAD — must modify this every time a new shape is added
def total_area(shapes):
    total = 0
    for s in shapes:
        if isinstance(s, Circle): total += s.r**2 * 3.14
        elif isinstance(s, Rect): total += s.w * s.h
    return total

# ✅ GOOD — just add a new Shape subclass to extend
def total_area(shapes):
    return sum(s.area() for s in shapes)  # closed for modification!
L — Liskov Substitution Principle

Objects of a subclass must be replaceable for objects of the superclass without breaking the program. A child class should honor the parent's contract.

Python
# ❌ BAD — Square breaks Rectangle's contract (w and h must be independent)
class Rectangle:
    def set_width(self, w): self.w = w
    def set_height(self, h): self.h = h

class Square(Rectangle):   # ❌ violates LSP!
    def set_width(self, w): self.w = self.h = w  # changes both!

# ✅ GOOD — Keep them as separate classes; both extend Shape
I — Interface Segregation Principle

Clients shouldn't be forced to implement interfaces they don't use. Many small interfaces beat one fat interface.

D — Dependency Inversion Principle

High-level modules should not depend on low-level modules. Both should depend on abstractions.

Python
# ❌ BAD — high-level depends on low-level directly
class Notification:
    def send(self, msg):
        EmailService().send_email(msg)  # tightly coupled!

# ✅ GOOD — depend on abstraction
class MessageSender(ABC):
    @abstractmethod
    def send(self, msg): ...

class Notification:
    def __init__(self, sender: MessageSender):
        self.sender = sender    # injected dependency!
    def send(self, msg):
        self.sender.send(msg)

Design Patterns
Reusable solutions to common problems

Design patterns are proven solutions to recurring design problems. The "Gang of Four" (GoF) book identified 23 patterns in 3 categories:

CategoryConcernKey Patterns
CreationalHow objects are createdSingleton, Factory, Builder, Prototype
StructuralHow objects are composedAdapter, Decorator, Facade, Proxy
BehavioralHow objects communicateObserver, Strategy, Command, Iterator
Python — Singleton Pattern
class DatabaseConnection:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

db1 = DatabaseConnection()
db2 = DatabaseConnection()
print(db1 is db2)  # True — same instance!
Python — Observer Pattern
class EventSystem:  # Subject / Publisher
    def __init__(self):
        self._listeners = {}

    def subscribe(self, event, callback):
        self._listeners.setdefault(event, []).append(callback)

    def emit(self, event, data=None):
        for cb in self._listeners.get(event, []):
            cb(data)

events = EventSystem()
events.subscribe("user_login", lambda u: print(f"Welcome, {u}!"))
events.subscribe("user_login", lambda u: print(f"Audit: {u} logged in"))
events.emit("user_login", "Alice")
# Welcome, Alice!
# Audit: Alice logged in
Python — Decorator Pattern
from functools import wraps

def log_call(func):   # Decorator adds behavior without changing func
    @wraps(func)
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}...")
        result = func(*args, **kwargs)
        print(f"Done. Result: {result}")
        return result
    return wrapper

@log_call
def add(a, b): return a + b

add(3, 4)
# Calling add...
# Done. Result: 7

Advanced OOP Concepts
Metaclasses, mixins, dataclasses & more

Mixins

A mixin is a class that provides methods to be used by other classes, without being a "parent" in the traditional sense. They add capabilities via multiple inheritance.

Python — Mixins
class JSONMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class TimestampMixin:
    def created_at(self):
        from datetime import datetime
        return datetime.now().isoformat()

class User(JSONMixin, TimestampMixin):  # mixed in!
    def __init__(self, name, email):
        self.name = name
        self.email = email

u = User("Alice", "alice@example.com")
print(u.to_json())      # {"name": "Alice", "email": "alice@example.com"}
print(u.created_at())   # 2025-01-15T10:30:00

Dataclasses

Python — @dataclass
from dataclasses import dataclass, field

@dataclass
class Product:
    name: str
    price: float
    tags: list = field(default_factory=list)

    # __init__, __repr__, __eq__ auto-generated!

p = Product("Laptop", 999.99, ["electronics", "tech"])
print(p)  # Product(name='Laptop', price=999.99, tags=['electronics', 'tech'])

Metaclasses

A metaclass is the class of a class — it controls how classes themselves are created. "When Python sees class Dog, it uses a metaclass to build the Dog class."

Python — Metaclass
class SingletonMeta(type):  # inherits from `type` (the metaclass of all classes)
    _instances = {}

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Config(metaclass=SingletonMeta):
    def __init__(self): self.settings = {}

c1 = Config(); c2 = Config()
print(c1 is c2)  # True
